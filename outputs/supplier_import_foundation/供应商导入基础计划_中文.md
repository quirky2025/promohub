# 供应商导入基础计划

这份中文说明是给业务决策和执行顺序看的。英文 SQL / CSV 文件保留英文，是因为数据库字段、程序字段和规则表更适合英文。

## 一句话总结

新的供应商产品，最后会变成和现在 TRENDS 一样的统一 QuirkyPromo 产品数据；但是第一步不能直接写进 `products`，必须先完整进入 raw staging 表，检查通过后再转换。

## TRENDS 和新供应商是什么关系

TRENDS 现在可以在线展示，说明它已经是整理好的成品数据。前台最终读取的是统一结构，例如：

- `products`
- `product_colours`
- `pricing_tiers`
- 产品图片
- category / subcategory
- MOQ
- supplier / supplier_sku
- slug

新的四家供应商最终也要变成这个统一结构。区别是：新供应商的原始文件格式各不相同，所以不能一开始就硬塞进 TRENDS 现在的成品结构。

## 为什么先做 raw staging

raw staging 的意思是：先把供应商原始数据原样保存下来，不急着整理、不急着发布、不急着变成产品。

这样做是为了保护：

- 原始 supplier_sku
- 原始产品名
- 原始分类路径
- 原始价格行
- 原始颜色
- 原始图片
- 颜色和图片之间的关系
- decoration / branding 选项
- MOQ / lead time / fulfillment 信息
- 供应商原始 JSON 或原始字段

如果一开始就直接转 `products`，最容易出错的是颜色和图片。例如蓝色产品图片被放到红色颜色下面，这种错误前台很难自动发现，但客户看到会很明显。

## 推荐导入流程

```text
supplier 原始文件
  ↓
raw staging tables
  ↓
READONLY 检查
  ↓
mapping rules / brand aliases / manual review
  ↓
transform preview
  ↓
人工确认
  ↓
products + product_colours + pricing_tiers + decoration options
```

## staging 表大概做什么

`supplier_import_batches`  
记录每一次导入是哪家供应商、哪个文件、多少行、当前状态。

`supplier_raw_product_rows`  
保存供应商原始产品行。重复 SKU 不删，先保留。

`supplier_raw_colour_options`  
保存原始颜色。颜色名、颜色 code、排序都先保留。

`supplier_raw_images`  
保存原始图片，并保留它对应哪个颜色。

`supplier_raw_price_rows`  
保存原始价格行。尤其 Logoline 这种一个 SKU 多个价格 / decoration 行的供应商，不能直接去重删行。

`supplier_raw_decoration_options`  
保存印刷、刺绣、logo 区域、setup cost、run cost 等 decoration 信息。

`supplier_transform_preview`  
这是最重要的审核表。它不是最终产品，而是告诉我们：这个 SKU 准备导到哪个 category/subcategory，有没有 warning，需不需要人工确认。

## 颜色和图片保护规则

这部分最重要。

- 不要把所有图片直接合并成产品图片列表。
- 如果供应商给了颜色图片关系，必须保留。
- 如果图片没有颜色关系，先标记为 unlinked，不要猜。
- 如果颜色没有图片，保留颜色并标记 missing image。
- 如果同一张图片对应多个颜色，要明确保存关系，不要只选一个。
- 产品发布前，颜色图片关系必须检查。

## supplier_sku 去重规则

- `supplier_sku` 只在同一个供应商内部唯一。
- 不同供应商出现同一个 SKU，不一定是同一个产品。
- 同一个供应商同一个 SKU 多行，可能是价格、颜色、decoration 重复行，不要直接删除。
- 如果同一个供应商同一个 SKU 对应多个产品名，要 block，人工看。

## 分类 mapping 原则

当前可沿用这些已有成果：

- `supplier_category_mappings_DRAFT_v4.csv`
- `supplier_keyword_rules_DRAFT.csv`
- `remaining_review_keyword_rules_DRAFT.csv`

但是要坚持保守原则：

- 高置信度才自动 mapping。
- 混杂产品继续 manual review。
- `Sale`、`Trending`、`Eco+ Collection` 是 collection/tag，不是 category。
- `Offshore Express`、`Indent` 是 fulfillment，不是 category。
- Gift Set / Kit / Pack / Hamper 不做普通 category，进入 Kits/Bundles 逻辑。

## 第一家 pilot 建议

建议第一家用 Gear For Life。

原因：

- 数据量较小
- 分类比较清楚
- 重复价格 / decoration 行相对没那么复杂
- 适合先验证 staging + health check + transform preview 这条流程

暂时不建议第一家就做：

- Logoline：行数多，价格和 decoration 重复行复杂。
- NIConcept：很多 broad category，需要 keyword mapping。
- PromoBrands：多 category path、collection、Offshore Express 信号比较多。
